# Data and Storage

## Data categories
- Source corpus: PDFs in `pdfs/`.
- Metadata: `Paperpile.json`.
- Graph database: `db/data` (Neo4j).
- Parse cache: `.cache/rag_articles/*.pkl` keyed by file stat + metadata + chunk settings.
- Legacy/aux artifacts: `upload.pkl`, `chunks_df.pkl`, `pdfs.pkl`, spreadsheets, citation text exports.

## Ingest-related files
- `Paperpile.json` drives metadata enrichment and ingest eligibility.
- `missing_metadata_pdfs.csv` can be generated by script for coverage gaps.

## Storage risks
- Very large local footprint (tens of GB) increases backup, sync, and scan time.
- `db/*` permission split (`7474` owner in several dirs) can cause host-level access friction.
- Binary artifacts are mostly gitignored; local state may diverge across machines.

## Permission notes
`db/conf`, `db/plugins`, `db/import`, `db/backups` were non-readable for non-owner during review.

## Backup/restore hooks
- `backup_neo4j.sh` uses `neo4j-admin database dump/load` via `neo4j-admin` container.
