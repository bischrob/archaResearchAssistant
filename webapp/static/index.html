Research Assistant RAG

Neo4j-backed PDF ingestion and contextual retrieval.

How To Use

  1. Click Sync PDFs to mirror Google Drive Paperpile files into local pdfs/.
  2. Choose ingest mode: test3 for partial ingest (first N using Partial Count), all for full corpus, or custom with explicit file paths.
  3. By default, already ingested PDFs are skipped. Enable Override Existing to reprocess them.
  4. Run Ingest to add/update graph data. Existing Neo4j data is never erased. Unreadable files are skipped and shown in failed_pdfs.
  5. Use Stop on Sync/Ingest/Query to cancel a running action.
  6. Ask a retrieval query in Search to get chunks with author and citation context.

Architecture

  • Data source: PDFs in pdfs/ plus bibliographic metadata from Paperpile.json.
  • Graph model: Article, Author, Chunk, Token, Reference, CITES.
  • Retrieval: Hybrid token + vector search, then lexical reranking.
  • LLM grounding: RAG chunks are passed as citation blocks [C1], [C2], etc.; answers are expected to cite these IDs inline.
  • Safety: citation enforcement + unsupported-sentence risk audit.
  • Incremental ingest: already-ingested files can be skipped; parsing cache is used for speed.

How Query Works

  1. Pre-LLM query parsing: the query is parsed into likely terms (tokens, years, quoted phrases, author-like terms).
  2. Candidate retrieval: the parsed query is matched through multiple channels: chunk vectors, token overlap, and author-node matches.
  3. Relationship expansion: each chunk is connected to its parent Article and related Author nodes, and article citation links are inspected through (Article)-[:CITES]->(Article).
  4. Context enrichment: results include citation neighborhood signals (what the source cites and what cites it) so highly connected, contextually relevant work is easier to surface.
  5. Reranking: candidates are reranked using chunk/title/author overlap, year and phrase matches, and graph-aware features to produce a final combined score.
  6. Low-confidence fallback: if top results do not strongly match title/author signals, author-channel candidates are force-merged and reranked before final context is selected.
  7. Grounded answering: selected chunks are passed to the LLM as numbered citation blocks ([C1], [C2], ...), and final answers are expected to cite those blocks inline.

Relationship usage summary: Author -[WROTE]-> Article -[HAS_CHUNK]-> Chunk, Article -[CITES]-> Article, and reference metadata links are used to provide richer retrieval context than isolated chunk matching.
